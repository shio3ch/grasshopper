## Cloudflare Workers + better-authで発生した「hanging promise」問題

Cloudflare Workers上でbetter-authを利用しているアプリケーションで、特定ユーザーのリクエストだけが応答待ちのまま固まり、ブラウザの再起動でしか回復しないという現象が報告されました。原因はCloudflare Workers特有の「hanging promise」という仕様で、リクエスト処理中に生成されたPromiseはクライアントが切断すると永遠に未解決のまま残ります。better-authは複数箇所でPromiseを遅延キャッシュしており、初期化を担当したリクエストが中断されると、そのisolate内の認証処理全体が恒久的にハングしてしまいます。同じクライアント接続は同じisolateにルーティングされ続けるため、被害が特定ユーザーに偏っていたことも判明しました。

エッジランタイム特有の実行モデルが原因で発生する不具合は、ローカル環境では再現しづらく発見が困難です。対策として、AsyncLocalStorageの事前セットによる遅延解決経路の回避、`ctx.waitUntil`への処理の係留、Promise.raceによるハング検知と自己修復という3層の防御を実装し、無応答の頻発を解消したとのことです。Cloudflare Workersなどのサーバーレスエッジ環境で認証ライブラリを使う開発者にとって、参考になる事例です。

詳細は [Cloudflare Workers + better-auth で全リクエストが無応答になる - hanging promise の罠](https://zenn.dev/coji/articles/cloudflare-workers-better-auth-hanging-promise) を参照。

## Claude Code (Opus 4.8) が数分固まる問題、188セッション実測の結果

Claude Code (Opus 4.8) を使用中にツールが数分間フリーズしたように見える現象について、188セッション・約14,000ターン分のJSONLログを解析した調査記事が公開されました。ユーザー入力から応答までの経過時間（レスポンスギャップ）を算出し、その間の出力トークン生成レートからシステム状態を分類する手法が取られています。

当初はAPI待機やネットワーク遅延が原因と予想されていましたが、実際には60秒を超える停滞376件のうち、真にAPI待機だったのはわずか1件で、残り375件は継続的なトークン生成が起きていたことが判明しました。真の原因は「拡張思考（extended thinking）」の生成にあり、複雑な依頼を受けるとOpus 4.8は内部推論を含め数千〜数万トークンを1ターンで生成するため、生成速度自体は通常通りでも総量の多さから数分を要し、その間ユーザーには視覚的なフィードバックが表示されないというものでした。Sonnet 5と比較してOpus 4.8は120秒超の停滞が3.4倍多く発生しており、これは不具合というよりも複雑なタスクに対してより深く推論する特性の表れだと分析されています。対策としては、探索作業をサブエージェントに委譲する構造化されたワークフロー設計が提案されています。

詳細は [Claude Code (Opus 4.8) が数分固まる問題、188セッション実測したら原因はAPIでもネットワークでもなかった](https://zenn.dev/yuki_fujisawa/articles/a155d388e61acc) を参照。

## AIエージェント時代の新ターミナルマルチプレクサ「herdr」

複数のAIコーディングエージェントを同時に管理するために設計されたRust製ターミナルマルチプレクサ「herdr」が話題になりました。2026年3月に公開されて以来、約1万スターを獲得しています。著者はtmuxを長年使ってきましたが、Claude Code・Codex・Cursor・Kiroなど複数のエージェントを同時運用する中で、「どのエージェントが承認待ちか確認するには各ペインを順番にチェックする必要がある」というtmuxの限界に直面し、乗り換えを決めたとのことです。

herdrはBlocked・Working・Done・Idle・Unknownという5つのエージェント状態を、設定不要でプロセス監視と出力パターンマッチングを組み合わせて自動検出します。デフォルトのプレフィックスキーはtmuxと同じ`Ctrl+B`で、経験者の学習コストをほぼゼロに抑えている点も特徴です。さらにCLI・ソケットAPIを備え、エージェント自身がペイン分割やコマンド実行、特定の出力待ちなどを自律的にオーケストレーションできる設計になっています。複数エージェントの並行運用が一般化する中で、こうした専用ツールの需要が高まっていることがうかがえます。

詳細は [AIエージェント時代のターミナルマルチプレクサ「herdr」にtmuxから乗り換えた](https://zenn.dev/studypocket/articles/herdr-ai-agent-multiplexer) を参照。

## 「自分の仕事はloopを書くこと」——Loop Engineeringが4800スターを獲得

AIを手動でプロンプトするのではなく、実行・評価・改善を自動的に繰り返すシステムを設計するという開発スタイル「Loop Engineering」が注目を集めています。Claude Codeのリーダーを務めるBoris Cherny氏の「自分の仕事はloopを書くことだ」という発言がこの概念を広めるきっかけとなりました。この考え方を実践に落とし込んだ`cobusgreyling/loop-engineering`リポジトリは、`loop-init`・`loop-audit`・`loop-cost`といった実用的なCLIツールを提供し、4800スターを獲得しています。

Loop Engineeringでは、AIを単発の応答を返すツールとしてではなく、スケジュールされたトリガー・ゴールベースの完了条件・作業の衝突を防ぐ隔離されたワークスペース・組み込みの検証機構・セッションをまたぐ永続的なメモリ・外部システムとの連携を備えた自律エージェントとして扱います。一方で記事は、この手法が開発の意思決定速度を増幅させる反面、「理解の負債（comprehension debt）」——コードが人間の理解速度を超えて蓄積してしまう問題——を生む危険性も指摘しており、監督と理解を放棄しないことが成功の鍵だと結論付けています。

詳細は [すでに4800スター、Loop Engineeringとは — "自分の仕事はloopを書くこと"](https://zenn.dev/acntechjp/articles/0c63b5b08bbdb9) を参照。

## Goのゼロコピー最適化、ラップした瞬間に消える「速い経路」

Go言語の標準ライブラリが`sendfile(2)`や`splice(2)`システムコールを介して自動的に行うゼロコピー最適化について、実測を交えて解説する記事が話題になりました。`io.Copy(conn, file)`を呼び出すと、コピー先が転送元を`*os.File`と認識できる場合に限り、ユーザー空間にデータを展開せずページキャッシュからソケットへ直接転送する内部関数が呼ばれます。

著者が512MiBのファイル転送で検証したところ、素の`*os.File`を渡した場合は約2,958回の`sendfile`呼び出しで完了する一方、汎用的なreaderでラップしただけで最適化が外れ、約131,000回もの`read`/`write`ペアにフォールバックしCPU使用量が3.4倍に増加しました。ロギングやトレーシングのための一見無害なラッパーが、この高速パスを静かに無効化してしまう点は多くの開発者が見落としがちな落とし穴であり、パフォーマンスが重要なファイル配信処理を書く際には注意が必要です。

詳細は [Zero-copy in Go: sendfile, splice, and the cost of io.Copy](https://segflow.github.io/post/zero-copy-sendfile-splice/) を参照。

## 「依存関係はVCSから直接取得すべき」という主張

Hacker Newsで、npmやRubyGemsのような中央集権的なパッケージレジストリへ公開する方式よりも、Goのように依存関係をバージョン管理システム（VCS）から直接取得する方式の方が優れているという主張がまとまった記事が注目を集めました。著者は、パッケージ公開モデルではソースコードと配布物の間に乖離が生まれ、実際にインストールされる内容を監査しづらいというセキュリティ上の問題を指摘しています。

VCSベースの配布であれば、コミット履歴をそのまま確認でき、更新前に実際の変更差分をレビューすることが容易になります。対照的にRubyGemsなどのパッケージを監査するには、ダウンロード・展開・手動での差分比較という手間のかかる作業が必要で、これが十分な監査を妨げているとのことです。サプライチェーン攻撃が相次ぐ中、依存関係の配布方式そのものを見直すべきだという議論は、パッケージマネージャの設計を考える上で示唆に富みます。

詳細は [Dependencies should be fetched directly from VCS](https://www.arp242.net/deps-vcs.html) を参照。

## Flipper Zero、開発体制を刷新——GitHub Discussionsでの投票制へ

Flipper Devicesチームが、Flipper Zeroのファームウェア開発体制を刷新することを発表しました。2024年に安定版1.0をリリースし動的アプリロードでメモリ制約に対応して以降、チームはインフラ保守と重大バグ修正に絞って対応し、リソースの多くを新デバイス開発に振り向けてきたとのことです。

ユーザー数が100万人を超えたことで、リアルタイムのコミュニティ対応が事実上不可能になったとチームは説明しており、今後は機能リクエストをGitHub Discussionsでの投票制に一本化し、開発者間のやり取りも非同期のみで行う方針です。プルリクエストの審査も厳格化され、特にAI生成コードへの精査が強化されるほか、全ファームウェア変更に統合テストが必須化されます。最も投票数の多いリクエストは毎週レビューされる予定で、Kickstarterでの公約はすべて果たした上で、コミュニティが独自にファームウェアやアプリを開発できるプラットフォームは維持されるとしています。

詳細は [The future of Flipper Zero development](https://blog.flipper.net/future-of-flipper-zero-development/) を参照。
